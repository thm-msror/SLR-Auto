Use this context when drafting the methodology section of a paper that used ATLAS. The goal is not to describe the repository as software for its own sake, but to explain the review procedure in a way that is academically defensible, reproducible, and aligned with what the current implementation actually does.

ATLAS should be described as a human-guided, API-based systematic literature review workflow. The review begins with researcher-defined research questions entered into the system. ATLAS then uses a GPT model to propose a Boolean search expression derived from those questions. Importantly, this Boolean query is not used automatically without oversight: the researcher is expected to review and, if necessary, edit the generated query before search execution. The approved Boolean expression is then expanded into several shorter executable search strings so that retrieval can be performed across the supported source APIs with better precision and compatibility.

When writing the database selection subsection, state only the databases that the current implementation truly searches as primary discovery sources: IEEE Xplore, Crossref, and Semantic Scholar. These are the core retrieval endpoints used by the pipeline. OpenAlex may be described as a metadata enrichment and open-access lookup source, but not as a primary bibliographic database searched in the initial identification stage. ACM Digital Library, Scopus, and Web of Science should not be claimed as directly searched databases for ATLAS unless the specific study added separate manual steps outside the codebase. A careful wording is that ATLAS queried IEEE Xplore, Crossref, and Semantic Scholar, then enriched records with OpenAlex where available. If needed, it is also accurate to note that some ACM, Elsevier, Springer, or Nature papers may still appear in the corpus through Crossref indexing or later PDF retrieval pathways, but those publishers are not independent database endpoints in the default workflow.

When writing the search strategy subsection, explain that ATLAS supports a researcher-in-the-loop search protocol. The researcher defines the review topic through the research questions; the system converts those questions into a draft Boolean expression; the researcher validates or revises that expression; and the approved query is decomposed into multiple source-ready searches. This means the conceptual search strategy is researcher-led, while the operational query formulation is computationally assisted. If the paper needs stronger methodological language, describe ATLAS as combining LLM-assisted query generation with human validation prior to retrieval.

When writing the inclusion and exclusion criteria subsection, explain that ATLAS does not rely on one fixed set of criteria across all studies. Instead, it generates review-specific inclusion and exclusion criteria from the researcher’s question using a GPT prompt designed for SLR protocol construction. The resulting criteria are intended to be answerable using title, abstract, publication year, and publisher only. The criteria are typically phrased as explicit yes/no questions and include standard scholarly quality checks, such as whether the study was published in a peer-reviewed venue and whether it should be excluded as non-peer-reviewed material. However, these criteria are not final until reviewed by the researcher. In the paper, the most accurate description is that ATLAS proposed the screening criteria and the researchers confirmed or refined them before automated screening began.

When writing the screening process subsection, describe ATLAS as semi-automated rather than fully manual or fully autonomous. Human involvement occurs at the protocol-definition stage, where the researcher reviews the Boolean search and the screening criteria. After those checkpoints, the title and abstract screening stage is automated. ATLAS evaluates each paper against the approved criteria using GPT and restricts the decision context to title, abstract, year, and publisher. For each criterion, the model returns YES, NO, or INSUFFICIENT. The system then computes a relevance score as the number of inclusion criteria answered YES minus the number of exclusion criteria answered YES. Records with non-positive screening scores are treated as excluded at the initial screening stage, while higher-scoring records are prioritized for downstream analysis. In the current implementation, the downstream full-text eligibility stage is also automated: selected PDFs are downloaded, analyzed with a GPT file-based workflow, and classified for inclusion while evidence is extracted for thematic synthesis. Therefore, if the study used ATLAS as provided in this repository, the screening process should be described as human-guided at setup and automated at both title/abstract screening and full-text eligibility assessment.

When writing about search dates, do not describe them as a universal repository constant. ATLAS is a rerunnable system, so each review run stores its own timestamps in `data/runs/<run_id>/log.json`, especially `created_at` and `updated_at`. The methodology section for a paper should therefore report the search dates from the specific ATLAS run used in that study. If the paper needs exact dates, they must be copied from the relevant run log rather than invented from the repository history. In the local copy of this project, example run folders span 2026-03-23 to 2026-03-26, but these dates are only examples of previous executions, not default search dates for all studies.

If the paper requires a concise methodology framing statement, a safe formulation is: the review used ATLAS as a human-guided automated SLR pipeline in which researchers defined the research questions, validated the Boolean search strategy and screening criteria, and then used the system to retrieve records from IEEE Xplore, Crossref, and Semantic Scholar, perform automated title-and-abstract screening, conduct automated full-text eligibility assessment on retrieved PDFs, and maintain PRISMA-aligned record counts throughout the workflow.

Do not overclaim capabilities that are not implemented. Do not state that ATLAS directly searched Scopus, Web of Science, or ACM Digital Library unless that was done outside the tool. Do not state that all screening decisions were manual if the built-in GPT screening pipeline was used. Do not present the inclusion and exclusion criteria as static repository defaults if they were generated from the study’s research question. Prefer wording that distinguishes between researcher approval and automated execution.
